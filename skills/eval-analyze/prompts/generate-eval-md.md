You are generating an eval.md file that caches the skill analysis so it doesn't need to be repeated. It also serves as readable documentation for the evaluation setup.

## Input

You will be given:
1. The structured YAML analysis from the analyze-skill step — a YAML block with `purpose`, `inputs`, `outputs`, `sub_skills`, `flags`, `pipeline`, `quality_criteria`, and `suggested_judges` fields, followed by a narrative
2. The skill name and SHA256 hash prefix of its SKILL.md (the executor computes the hash — use the value provided, don't compute it yourself)

## Output Format

Write a complete eval.md file with two parts:

### Part 1: YAML Frontmatter

Between `---` markers:

```yaml
---
# Auto-generated by /eval-analyze — edit to override
skill: <skill_name>
analyzed_at: <ISO 8601 timestamp>
skill_hash: <SHA256 hash prefix of SKILL.md, 12 chars>

# Discovered skill capabilities
headless: <true/false>
dry_run: <true/false>

# Suggested judges (summary from analysis)
suggested_judges:
  - name: <judge_name>
    type: <check|llm>
    description: "<what this judge checks>"
---
```

### Part 2: Markdown Body

Write a clear, concise narrative covering:

1. **Skill Analysis** — what the skill does, in human terms
2. **Inputs** — what the skill expects (how cases are structured, what files and fields)
3. **Outputs** — what files the skill creates, their structure and naming
4. **Pipeline Flow** — what happens in what order (step by step)
5. **Quality Criteria** — what "good" looks like for this skill's output

Write for a developer who needs to understand the evaluation setup at a glance. Reference actual file paths, field names, and criteria you observed. Don't be generic — if you saw `input.yaml` with a `prompt` field, say so.
